# spec-graph checklist

为 `plan/story/*` 生成实现前质量检查清单。

## Architecture Principle

**spec-graph 不判断 story 是否"足够好" — 只机械扫描并标记问题。**

- ❌ spec-graph 不会替你写 AC
- ❌ spec-graph 不会替你拆分过大的 story
- ❌ spec-graph 不会替你消除模糊形容词
- ❌ spec-graph 不会替你识别所有边界情况
- ✅ spec-graph 机械扫描 story 文档 + trace 数据,生成 checklist
- ✅ spec-graph 自动检测 13 种模糊形容词(fast, robust, scalable, ...)
- ✅ spec-graph 把 checklist 写到 `.spec-graph/checklists/<story-id>.md`,等 agent/user 手动 review soft check
- ✅ 5 个 mechanical check 自动 pass/fail,5 个 soft check 需手动标记

**Agent 的职责**:跑 checklist → 读 mechanical 失败项 → 修文档(补 AC、拆 story、消除模糊词)→ 手动 review soft check → 标记完成。

## What this does

对每个 `plan/story/*` artifact,生成 10 项检查清单:

### 5 Mechanical Checks(自动验证)

| # | Check | 验证方式 | 失败原因 |
|---|-------|---------|---------|
| 1 | Story references at least one requirement | 查 trace `derives` 边是否指向 requirement | 没建 trace,或 trace 指向非 requirement |
| 2 | Scope is atomic (implementable in one iteration) | 文档行数 ≤ 200 | 文档过长(>200 行),应拆分 |
| 3 | Has at least 2 acceptance criteria | 正则匹配 `^### AC-\d+` | AC 少于 2 个 |
| 4 | All referenced requirements are resolved | 检查 derives 目标是否 completed | 引用的 requirement 还在 pending |
| 5 | No file paths outside project scope | 检测 `/etc/`, `/usr/`, `/var/`, `/tmp/`, `~/` | 引用了系统路径或越界路径 |

### 5 Soft Checks(需手动 review)

| # | Check | 自动辅助 |
|---|-------|---------|
| 1 | No ambiguous adjectives | **自动检测** 13 种词:fast, quick, user-friendly, intuitive, robust, reliable, scalable, flexible, simple, easy, seamless, performant, secure |
| 2 | Each AC is verifiable by automated test | 提示:确保 AC 能用 unit/integration test 验证 |
| 3 | Edge cases considered and documented | 提示:加 edge case 到 AC 或单独 section |
| 4 | Dependencies declared | 提示:加 "Depends On" section |
| 5 | Out-of-scope items explicitly listed | 提示:加 "Out of Scope" section 防 scope creep |

> **注**:如果 story 文档不存在(mechanical check 跳过 = passed,soft check 中 ambiguous = null)。所以最好在文档生成后再跑 checklist。

## Usage

```bash
# 为某个 story 生成 checklist
spec-graph checklist plan/story/user-login

# JSON 输出(供脚本消费)
spec-graph checklist plan/story/user-login --json
```

### Options

| Option | Description |
|--------|-------------|
| `--json` | 输出 JSON:`{story_id, checklist_path, mechanical_checks, soft_checks}` |

### Positional Argument

| Argument | Required | Description |
|----------|----------|-------------|
| `<story-id>` | ✅ Required | Story 的 artifact ID,如 `plan/story/user-login` |

- [x] **Has at least 2 acceptance criteria**
- [x] **All referenced requirements are resolved**
- [x] **No file paths outside project scope**

## Soft Checks (Manual Review)

## 13 种模糊形容词(自动检测)

| 词 | 建议 |
|----|------|
| `fast` | 指定可测量目标(如 `<200ms`) |
| `quick` | 指定可测量目标 |
| `user-friendly` | 列出具体 UX 要求 |
| `intuitive` | 描述预期用户行为 |
| `robust` | 指定错误处理要求 |
| `reliable` | 指定 uptime/availability 目标 |
| `scalable` | 指定预期负载/规模 |
| `flexible` | 描述具体扩展点 |
| `simple` | 描述最小必需步骤 |
| `easy` | 指定可测量标准 |
| `seamless` | 描述集成要求 |
| `performant` | 指定 latency/throughput 目标 |
| `secure` | 列出具体安全要求 |

> 检测逻辑:扫描文档每一行(跳过 `#` 标题和 ``` 代码块),匹配即报行号 + 建议。

## 何时使用 — 判断标准

### ✅ 应该使用 checklist

| 场景 | 时机 |
|------|------|
| sub-agent 刚写完 story 文档 | 立即跑 checklist 验证质量 |
| story 状态从 draft → ready | 跑 checklist 作为门槛 |
| 进入 implement 阶段前 | 最后一次验证 |
| code review 时 | 给 reviewer 看 checklist |
| 多个 story 批量验证 | 循环跑每个 story |
| 发现 story 质量参差 | 用 checklist 标准化 |

### ❌ 不应该使用 checklist

| 场景 | 替代做法 |
|------|---------|
| 跨 artifact 一致性分析 | `spec-graph analyze` |
| 验证 gate 是否通过 | `spec-graph gate` |
| 跑测试 | `spec-graph check` |
| 修改 story 内容 | 直接编辑 `.spec-graph/artifacts/plan/story/*.md` |
| 非 story artifact | 此命令只支持 `plan/story/*` |

## Agent Workflow

```
1. sub-agent 写完 story 文档(通过 dispatch manifest)
   ↓
2. spec-graph artifact complete plan/story/<name> --status in_progress
   ↓
3. spec-graph checklist plan/story/<name>
   ↓
4. 读输出:
   ├── mechanical ✗ → 修文档:
   │   ├── 缺 REQ mapping → 加 References section + 建 trace
   │   ├── 行数 >200 → 拆分 story(创建新 change)
   │   ├── AC <2 → 补 AC
   │   ├── REQ 未 resolved → 先完成上游 requirement
   │   └── 越界路径 → 修文档移除
   ├── soft check 中 ambiguous 词 → 替换为可测量标准
   └── 其他 soft check → 手动 review 后在 .md 标记 [x]
   ↓
5. 修完后再跑 spec-graph checklist 确认 mechanical 全 ✓
   ↓
6. 标记 soft check 完成后,把 story 推进到 ready 状态
   ↓
7. 继续 dispatch loop
```

## 与 Agent 的协作关系

- **主 agent**:跑 checklist,读 mechanical 失败,分派修复
- **sub-agent (writer)**:接收 "fix story X" 任务,改文档
- **sub-agent (reviewer)**:可选 — review soft check(模糊词、边界情况等定性判断)
- **coordinator**:在 implement stage 前,可强制要求 checklist 全 pass
- **trace 系统**:mechanical check #1 和 #4 依赖 trace 数据(`derives` 边)

## Usage Scenarios

### Scenario 1: 标准 — 写完 story 后验证(成功)

```bash
# sub-agent 刚写完 plan/story/user-login.md
$ spec-graph artifact complete plan/story/user-login --status in_progress

$ spec-graph checklist plan/story/user-login
✓ Checklist generated: .spec-graph/checklists/plan_story_user-login.md

Mechanical Checks (automated):
  ✓ Story references at least one requirement
  ✓ Scope is atomic (implementable in one iteration)
  ✓ Has at least 2 acceptance criteria
  ✓ All referenced requirements are resolved
  ✓ No file paths outside project scope

Soft Checks (manual review):
  ○ No ambiguous adjectives
  ○ Each AC is verifiable
  ...

# agent 手动 review soft check,在 .md 中标 [x]
# 然后推进到 ready 状态
```

### Scenario 2: 失败 — 行数过多

```bash
$ spec-graph checklist plan/story/big-refactor
Mechanical Checks:
  ✗ Scope is atomic (implementable in one iteration)
    Story is 245 lines (max: 200). Consider splitting.
```

**修复**:
```bash
# 选项 A: 拆分 story(推荐)
spec-graph change create --title "Split big-refactor story" --type refactor
# 把 245 行拆成 2-3 个子 story

# 选项 B: 精简文档(把详细内容移到 design doc)
# 直接编辑 story.md,删除冗余
```

### Scenario 3: 失败 — AC 不足

```bash
$ spec-graph checklist plan/story/quick-fix
Mechanical Checks:
  ✗ Has at least 2 acceptance criteria
    Only 1 AC(s) found (minimum: 2).
```

**修复**:让 sub-agent 在 story.md 加更多 AC:
```markdown
### AC-1: 用户输入正确密码时登录成功
### AC-2: 用户输入错误密码时显示错误提示   ← 新加
### AC-3: 连续 5 次失败后锁定账户 30 秒   ← 新加
```

### Scenario 4: 失败 — 模糊形容词

```bash
$ spec-graph checklist plan/story/api-optimization
Soft Checks:
  ○ No ambiguous adjectives
    2 ambiguous adjective(s) found:
      L15: "fast" -> specify a measurable target (e.g., '<200ms')
      L42: "scalable" -> specify expected load/scale
```

**修复**:让 sub-agent 改文档:
```diff
- The API should be fast and scalable.
+ The API should respond in <200ms (p95) and handle 10K QPS.
```

### Scenario 5: 失败 — 缺 requirement mapping

```bash
$ spec-graph checklist plan/story/orphan-story
Mechanical Checks:
  ✗ Story references at least one requirement
    Add a "References" section listing requirement IDs
```

**修复**:
```bash
# 1. 在 story.md 加 References section
#    ## References
#    - requirement/prd/user-auth

# 2. 建 trace(或用 artifact complete 自动 wire)
spec-graph trace add --from plan/story/orphan-story --to requirement/prd/user-auth --relation derives

# 3. 重跑
spec-graph checklist plan/story/orphan-story
# ✓ Story references at least one requirement
```

### Scenario 6: 失败 — 引用未完成的 requirement

```bash
$ spec-graph checklist plan/story/feature-x
Mechanical Checks:
  ✗ All referenced requirements are resolved
    Resolve or remove unresolved requirement references
```

**修复**:
```bash
# 先把上游 requirement 推到 completed
spec-graph artifact complete requirement/prd/feature-x
# 然后重跑 checklist
```

### Scenario 7: 失败 — story 不存在

```bash
$ spec-graph checklist plan/story/nonexistent
✗ Story not found: plan/story/nonexistent
Available stories:
  - plan/story/user-login
  - plan/story/payment-flow
```

**修复**:用正确的 story ID(从 available 列表中选)。

## Error Handling

| 错误 | 原因 | 修复 |
|------|------|------|
| `Graph not found. Run spec-graph compose first.` | 没有 graph.yaml | 先 `spec-graph compose` |
| `Story not found: <id>` | story ID 错误或不存在 | 看输出的 available stories 列表 |
| `Story is already completed` | story 已经 done | checklist 是 pre-implementation 工具,后期无需跑 |
| mechanical 全 ✓ 但 soft 失败 | 文档质量不够 | 手动 review 并修文档,或标记 soft check 完成 |
| ambiguous 词检测不到 | 文档不存在或全在代码块内 | 确保文档已生成,模糊词在正文中 |

## 衔接关系

- **前置**:
  - `spec-graph compose`(必须有 graph.yaml)
  - `spec-graph prime`(必须有 machine-state.yaml)
  - sub-agent 已写完 story 文档(否则 mechanical 全跳过 = 假通过)
- **数据来源**:
  - `graph.yaml`(artifact 类型定义)
  - `machine-state.yaml`(story 状态、requirement 状态)
  - `.spec-graph/traces/*.yaml`(trace 数据,用于 REQ mapping/resolution 检查)
  - `.spec-graph/artifacts/plan/story/*.md`(story 文档内容,用于行数/AC/路径/模糊词检测)
- **输出**:`.spec-graph/checklists/<story-id>.md`
- **被引用**:
  - 进入 implement stage 前可作为强制门槛(由 coordinator 配置)
  - code review 时 reviewer 参考
- **配合**:
  - `spec-graph analyze`(跨 artifact 一致性,范围更广)
  - `spec-graph artifact complete`(标记 story 状态)
  - `spec-graph trace`(建 trace 边以通过 REQ mapping check)
