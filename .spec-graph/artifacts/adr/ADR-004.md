# ADR-004: 文档存储在 .spec-graph/artifacts/ 目录,spec-graph 只追踪元数据

## 状态

已接受

## 背景

spec-graph 需要管理 artifact 文档(如 PRD、架构文档、用户故事等)。

**约束条件**:
- spec-graph 是状态追踪引擎,不应成为内容管理系统
- AI Agent 负责生成文档内容,spec-graph 只追踪状态
- 文档需要版本控制,应该可以正常 git 管理
- 文档需要按类型组织,便于查找和管理

## 决策

采用 **分离存储策略**:
- **文档内容**: AI Agent 写入 `.spec-graph/artifacts/<type>/<id>.md`
- **元数据追踪**: spec-graph 在 machine-state.yaml 中记录 artifact 状态(pending/in_progress/ready/completed/failed/blocked)
- **链接追踪**: spec-graph 通过 analysis 命令记录文档路径、使用的模板、关联的任务

## 理由

1. **关注点分离**: spec-graph 专注状态追踪,内容由 AI Agent 生成
2. **灵活性**: AI Agent 可自由调整文档结构,不受 spec-graph 限制
3. **Git 友好**: 文档是普通 markdown 文件,可正常版本控制
4. **可追溯性**: 通过 analysis 命令建立任务→文档→模板的追踪链
5. **集中管理**: 所有文档存储在 .spec-graph/artifacts/ 下,与机器状态、分析记录保持一致

## 后果

### 正面
- spec-graph 保持简洁,不需要处理文件 I/O
- AI Agent 可自由生成和调整文档内容
- 文档可正常 git 管理,支持 diff/revert
- 通过 analysis 命令可追溯文档来源
- 所有 spec-graph 相关内容集中在 .spec-graph/ 目录下

### 负面
- AI Agent 需要负责写入文档,增加了复杂度
- 需要确保 AI Agent 正确更新 analysis 记录
- 文档路径需要标准化,避免混乱

## 替代方案

### spec-graph 存储文档内容
- 优势: 统一管理,简化 AI Agent 职责
- 劣势: spec-graph 变成内容管理系统,违背设计原则;大文件影响性能
- 结论: 不适合

### 文档分散存储(如项目根目录)
- 优势: 传统做法,符合习惯
- 劣势: 难以追踪,难以统一管理,与 spec-graph 状态分离
- 结论: 不适合
