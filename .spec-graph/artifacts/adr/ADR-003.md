# ADR-003: 采用三段式管线(Sense → Compose → Enforce)

## 状态

已接受

## 背景

spec-graph 需要从用户输入和现有 repo 生成工作流图,并在执行过程中强制质量门。

**约束条件**:
- LLM 只参与必要的分类任务,不应控制整个流程
- 工作流一旦合成就应冻结,可 diff/复现/审计
- 必须支持迭代变更(需求变化时重新合成)
- Gate 评估必须完全确定,不能有 LLM 临场判断

## 决策

采用 **三段式管线**:
1. **Sense(感知)**: LLM + 确定性扫描 → profile.yaml(9 维事实集)
2. **Compose(合成)**: 纯规则引擎,profile + pack 库 → graph.yaml
3. **Enforce(强制)**: 完全确定,等同 wdf-method 的 FSM/gate

## 理由

1. **LLM 最小化**: 只在 Sense 阶段使用 LLM 做分类,Compose 和 Enforce 完全确定
2. **可审计**: graph.yaml 是声明式产物,可 diff/版本化/审查
3. **可复现**: 相同 profile 总是生成相同 graph
4. **可迭代**: profile 可改,改 profile 触发重新 Compose,类似 OpenSpec 的 change

## 后果

### 正面
- LLM 错误被隔离在 Sense 阶段,不影响后续
- Compose 结果可审查和回滚
- Enforce 完全确定,行为可预测
- 支持需求变化时重新合成

### 负面
- Sense 阶段需要 LLM,增加了外部依赖
- Profile 需要人工复核后再冻结(避免 LLM 盲信)
- 重新 Compose 会覆盖 graph.yaml,需要版本管理

## 替代方案

### LLM 驱动整个流程
- 优势: 更灵活,可处理模糊需求
- 劣势: 不可复现,不可审计,行为不可预测
- 结论: 不适合,违背 spec-graph 的设计原则

### 完全静态(无 Sense 阶段)
- 优势: 完全确定
- 劣势: 用户需要手动定义 profile,使用门槛高
- 结论: 不适合,降低了易用性
