# ADR-001: 选择 TypeScript 作为实现语言

## 状态

已接受

## 背景

spec-graph 需要一个实现语言来构建 CLI 工具和工作流引擎。

**约束条件**:
- 需要跨平台支持(macOS/Linux/Windows)
- 需要与 wdf-method 的代码复用
- 需要强大的类型系统
- 需要良好的生态系统(CLI 框架、YAML 处理等)
- 团队需要熟悉该技术

## 决策

选择 **TypeScript 5.x** 作为实现语言。

## 理由

1. **与 wdf-method 一致**: wdf-method 已经使用 TypeScript,可以直接复用其 FSM/gate/traceability 实现
2. **类型安全**: TypeScript 的类型系统帮助捕获错误,降低 bug 率
3. **跨平台**: Node.js 运行时天然跨平台
4. **生态丰富**: Commander.js(CLI)、js-yaml(YAML 处理)、Vitest(测试框架)等成熟工具
5. **团队熟悉**: 团队已有 TypeScript 经验,学习成本低

## 后果

### 正面
- 可以直接复用 wdf-method 的已验证实现
- 类型系统帮助捕获潜在错误
- 丰富的生态系统加速开发
- 团队成员可快速上手

### 负面
- 需要 Node.js 运行时环境
- 编译步骤(需要 TypeScript 编译器)
- 包大小相对较大(Node.js 依赖)

## 替代方案

### Python 3.11+
- 优势: 脚本语言,无需编译,生态丰富
- 劣势: 类型系统弱,与 wdf-method 不兼容,团队不熟悉
- 结论: 不适合

### Rust
- 优势: 高性能,类型安全
- 劣势: 学习曲线陡峭,生态不如 Node.js 丰富
- 结论: 过度设计

### Go
- 优势: 编译为单一二进制,性能好
- 劣势: 与 wdf-method 不兼容,生态不如 Node.js 丰富
- 结论: 不适合

- spec-graph 技术栈: TypeScript + Vitest + Commander.js + js-yaml
