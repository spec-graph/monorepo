# Story Map: spec-graph

## 核心用户活动

### 1. 初始化项目
**目标**: 快速设置 spec-graph 工作流

**用户故事**:
- 作为开发者，我希望通过一行命令初始化项目工作流，以便快速开始使用 spec-graph
- 作为开发者，我希望自动分析项目结构并生成适配的 profile，以便获得最佳的工作流配置
- 作为开发者，我希望能够自定义 profile 覆盖，以便满足特定的项目需求

### 2. 工作流调度
**目标**: 明确下一步应该做什么

**用户故事**:
- 作为 AI Agent，我希望获取下一步动作的 manifest，以便知道应该生成什么文档或运行什么检查
- 作为 AI Agent，我希望 manifest 包含文档指导字段，以便知道使用什么模板和存储在哪里
- 作为开发者，我希望能够查看所有建议的动作，以便了解完整的工作流路径

### 3. 文档管理
**目标**: 创建和管理项目文档

**用户故事**:
- 作为 AI Agent，我希望根据模板生成文档，以便确保文档结构的一致性
- 作为 AI Agent，我希望将文档注册为 artifact，以便追踪文档的状态
- 作为开发者，我希望运行 checklist 验证文档质量，以便确保文档满足质量标准

### 4. 追溯管理
**目标**: 建立和维护追溯关系

**用户故事**:
- 作为 AI Agent，我希望建立 artifact 间的 trace 关系，以便支持影响分析
- 作为开发者，我希望查询完整的追溯链，以便理解 artifact 间的依赖关系
- 作为项目经理，我希望分析变更的影响范围，以便评估变更的风险

### 5. 质量门控
**目标**: 确保工作流质量

**用户故事**:
- 作为系统，我希望在关键节点评估 gate，以便阻止不合格的产出进入下一阶段
- 作为开发者，我希望查看 gate 评估结果，以便了解哪些条件未满足
- 作为项目经理，我希望监控 gate 通过率，以便评估项目质量

### 6. 阶段分析
**目标**: 记录和分析每个阶段

**用户故事**:
- 作为 AI Agent，我希望记录每个阶段的分析和决策，以便支持审计和回溯
- 作为开发者，我希望查看历史阶段的分析，以便理解项目的演进
- 作为项目经理，我希望分析阶段的效率和效果，以便优化工作流

### 7. 状态监控
**目标**: 了解当前工作流状态

**用户故事**:
- 作为开发者，我希望查看当前阶段和进度，以便了解项目状态
- 作为开发者，我希望查看所有 artifacts 的状态，以便了解哪些工作已完成
- 作为项目经理，我希望查看整体的项目健康度，以便识别风险

## 用户旅程地图

```
初始化 → 调度 → 文档创建 → 注册 → Trace 建立 → Checklist → Gate 评估 → 状态转移
  ↓        ↓        ↓         ↓        ↓           ↓          ↓           ↓
 init   dispatch  create   complete  trace add   checklist   gate      transition
```

## 优先级排序

### P0 (必须)
1. 初始化项目 (spec-graph init)
2. 工作流调度 (spec-graph dispatch)
3. 文档注册 (spec-graph artifact complete)
4. 状态监控 (spec-graph status)

### P1 (重要)
1. 文档创建 (模板系统)
2. Trace 建立 (spec-graph trace add)
3. Checklist 验证 (spec-graph checklist)
4. Gate 评估 (spec-graph gate)

### P2 (期望)
1. 阶段分析 (spec-graph analysis)
2. 自动执行 (spec-graph run)
3. 追溯查询 (spec-graph trace)

### P3 (可选)
1. 可视化工具
2. 高级分析
3. 自定义 pack
