# PRD: spec-graph 工作流编排引擎

## 问题陈述

当前 AI 辅助开发面临以下问题:
- AI 生成代码缺乏结构化流程控制,容易跳过验证步骤
- 需求→设计→实现→验证的追溯链断裂,难以追踪决策来源
- 多 agent 协作时上下文丢失,重复工作或遗漏依赖
- 项目规范(宪法)难以强制执行,质量标准不统一
- 变更管理无记录,无法审计历史决策

需要一个**领域中立的规格驱动图编排内核**,把工作流从"AI 凭记忆做"变成"声明式 + 状态机驱动"。

## 用户故事

### US-1: 初始化项目工作流
**作为** 开发者,
**我希望** 能够用一行命令分析项目并生成工作流图,
**以便** 快速获得适配当前项目类型的流程定义。

**验收标准:**
- Given 一个新项目, When 运行 `spec-graph init`, Then 生成 `.spec-graph/` 目录和 profile.yaml
- Given 项目有 package.json, When sense 分析, Then 正确识别为 Node.js 项目
- Given 用户指定 `--build=api`, When 初始化, Then profile 中 has_ui=none, boundary=api

### US-2: 调度下一步动作
**作为** AI Agent(Claude/Codex),
**我希望** 运行 `spec-graph dispatch` 获取下一步动作,
**以便** 知道当前应该生成什么文档、运行什么检查、转移到哪个阶段。

**验收标准:**
- Given 工作流未完成, When 运行 `dispatch --json`, Then 返回 actions[0] 包含 type/id/agent_id
- Given 动作是 produce_artifact, When 返回 manifest, Then 包含 template_ref/suggested_doc_path/document_guidance
- Given 动作是 run_check, When 返回 manifest, Then 包含 check_command 可直接执行
- Given 动作需要 sub-agent, When 返回 manifest, Then requires_sub_agent=true 并包含 agent_prompt_ref

### US-3: 追踪文档与状态
**作为** 开发者,
**我希望** spec-graph 追踪每个阶段生成的文档和决策,
**以便** 能够回溯任何代码变更的来源。

**验收标准:**
- Given AI Agent 生成了 PRD 文档, When 运行 `spec-graph analysis --docs <path>`, Then 记录文档路径
- Given 想查看 propose 阶段的分析, When 运行 `spec-graph analysis --phase propose`, Then 显示该阶段的关键决策和文档
- Given 想验证文档质量, When 运行 `spec-graph checklist <artifact-id>`, Then 返回机械检查和软检查结果

### US-4: 管理变更
**作为** 开发者,
**我希望** 每个实质性变更都有记录和审计,
**以便** 能够追踪项目演进和回滚。

**验收标准:**
- Given 想做新功能, When 运行 `change create --title "..."`, Then 生成 change 描述符
- Given change 已应用, When 完成工作, Then 运行 `change complete` 记录完成时间
- Given change 已完成, When 运行 `change archive`, Then 归档到 .spec-graph/changes/archived/

### US-5: 强制质量门
**作为** 项目经理,
**我希望** 关键阶段转换前必须通过质量检查,
**以便** 防止不合格的产出进入下一阶段。

**验收标准:**
- Given 设计阶段未完成 architecture 文档, When 尝试 transition 到 implement, Then gate 阻止并返回 missing_artifacts
- Given 契约版本漂移, When 检查 require_contracts_current, Then 返回 missing_contracts
- Given trace 缺失, When 检查 require_traces, Then 返回 missing_traces 并提供 trace_query 详情

## 非功能需求

### 性能
- `dispatch --json` 响应时间 < 100ms(95% 情况)
- 支持 100+ artifacts 的 graph 而无明显性能下降
- 测试套件全量运行 < 10 秒

### 可移植性
- Node.js 20+ 运行时
- 支持 macOS/Linux/Windows
- 无原生依赖(纯 TypeScript)

### 可扩展性
- Pack 系统支持第三方扩展
- 模板可自定义
- Profile 维度可扩展

### 可维护性
- 代码覆盖率 > 80%
- 6 原语设计,内核零领域词
- 完整的协议文档(coordinator-protocol.md 等)

## 约束条件

### 技术约束
- 必须使用 TypeScript
- 必须保持中立引擎原则(不调用 LLM API,不 dispatch sub-agent)
- 必须兼容 wdf-method 的 FSM/gate/trace 机制

### 架构约束
- 6 原语: Work-unit / Artifact / Contract / Check / Gate / Trace-edge
- 内核不出现任何领域词汇(web/api/frontend 等)
- 所有领域知识以 pack 形式提供

### 设计约束
- spec-graph 只追踪状态,不存储文档内容
- AI Agent 负责文档生成和持久化
- 通过 manifest 指导 AI Agent

## 超出范围

以下功能明确不在 spec-graph 范围内:
- 调用 LLM API 生成内容(由 coordinator/AI Agent 负责)
- Dispatch sub-agent(由 coordinator 负责)
- 存储 artifact 文档内容(由 AI Agent 写入项目文件系统)
- 特定领域逻辑(由 pack 提供)
- UI 界面(纯 CLI)

## 成功指标

### 定量指标
- 测试覆盖率 > 80%(当前: 484 个测试通过)
- dispatch 响应时间 < 100ms
- 内核零领域词(已验证)

### 定性指标
- 能够无损复刻 wdf-method 的 Web 流程(纸面已验证)
- 支持任意领域(嵌入式、Web、API 等)
- AI Agent 能根据 manifest 自主完成工作流

## 开放问题

1. **Pack 市场**: 是否需要中心化的 pack 仓库?
   - 决策: 首期不建,用户本地 packs/ 即可

2. **可视化**: 是否需要 graph 可视化?
   - 决策: 首期纯 CLI,后续可加

3. **多项目治理**: 是否支持 workspace 级 profile?
   - 决策: 首期单项目,后续扩展

## 附录

### 术语表
- **Work-unit**: 有状态的工作节点
- **Artifact**: 带类型的产物(PRD/架构/代码等)
- **Contract**: 有 producer/consumer 双边的 typed Artifact
- **Check**: 声明式校验(shell 命令或规则)
- **Gate**: 状态转移上的不变量条件
- **Trace-edge**: 任意节点→任意节点的 typed 边
